The TPSGL Scheduler Bug

root cause — ~72% of TPSGL failures (May 2026)
Schedules configured for 09:00 Europe/London were firing at 00:00 UTC instead. Operator laptops were asleep at midnight → tasks went to offline agents → cascading "agent offline" → queue.Drain infinite-retry amplification (agent/internal/queue/queue.go:100-108). Visible above as red ticks at hour 0 on the schedule lane.
amplifier 1 — assigned-status zombie
expireOverdueTasks in tasks.service.ts:1044 excluded 'assigned' status — tasks zombie for 7d until TTL sweep, blocking re-issue.
amplifier 2 — queue.Drain infinite retry
When the backend returns 400 for a bad result payload, the agent re-queues forever rather than dropping. Combined with the timezone bug, a single bad task fires every poll cycle.

Capacity Math (rough)

load per hour (single primary)
• heartbeat-detect + alert-eval: 60 ticks/hr × 1 SQLite SELECT ≈ 60 queries/hr
• scheduler: 60 ticks/hr × 1 SQLite SELECT ≈ 60 queries/hr
• auto-rotation: 60 ticks/hr × 1 SQLite SELECT ≈ 60 queries/hr
• ingestion retry: 12 ticks/hr × N pending tasks via bulk ≈ 12 ES bulks/hr
• defender alerts pass: 12 ticks/hr × paginated Graph fetch + msearch + bulk ≈ 12 Graph rounds/hr
• defender scores: 0.17 ticks/hr (every 6h)
• heartbeat prune: 1 tick/hr
— heartbeat traffic itself is N agents × server-driven cadence (typically 30-60s) — agent-side, not primary-side
